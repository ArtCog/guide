# Видеомонтаж — Профессиональный скилл для Claude Code

> Скилл для работы с видео через Claude Code: удаление пауз, склейка, субтитры, оверлеи, финальный рендер. Проверено в боевых условиях на реальных YouTube-проектах.

---

## 0. Дерево решений — начни отсюда

```
Запрос пользователя → что делать:

"убери паузы / silence"       → C1. Удаление тишины (только auto-editor)
"вырежи фразу / cut phrase"  → C2. Вырезание фразы (транскрипция → trim+concat через NVENC)
"склей / concat"              → C3. Склейка (NVENC ре-кодирование, НИКОГДА не -c copy)
"субтитры / captions"         → C4. Субтитры
"текст на экране"             → C5. Текстовые оверлеи
"картинка / скриншот"         → C6. Изображение / B-Roll
"круглая голова / circle"     → C7. Круглая вебкамера
"зум / zoom"                  → C8. Динамический зум
"скорость / speed"            → C9. Скорость
"аудио / звук / voice"        → C10. Аудио-операции
"громкость / normalize"       → C11. Нормализация (loudnorm, ПОСЛЕДНИЙ шаг)
"цвет / color / cinematic"    → C12. Цветокоррекция
"вирусные субтитры"           → C13. Вирусный стиль субтитров

ЛЮБОЕ редактирование → Шаг 1: проверить транскрипт. Нет — сначала транскрибировать.
```

## 1. КРИТИЧЕСКИЕ ПРАВИЛА — читать первым делом

### 1.0 ИССЛЕДОВАТЬ перед повторной попыткой
Если что-то падает или даёт плохой результат **дважды** — СТОП. НЕ пробовать в третий раз вслепую.
Идём в интернет: GitHub issues, Stack Overflow, Reddit. Решение уже существует.
Пример: артефакты audio в auto-editor → GitHub issue #492 → нашли причину и фикс за одним поиском.
Никогда не изобретай велосипед. Сначала исследование, потом код.

### 1.1 НИКОГДА не использовать `-c copy` (stream copy)
Stream copy режет ТОЛЬКО на ключевых кадрах (каждые 2-4 секунды). Если точка разреза между ключевыми кадрами → **замороженные кадры на несколько секунд**. Это ВСЕГДА даёт видимые артефакты.
**ВСЕГДА ре-кодировать через NVENC.** Качество важнее скорости, без исключений. RTX-серия ре-кодирует 25 мин примерно за 3 мин.

### 1.2 НИКОГДА не использовать фильтры select/aselect для нарезки
Они вызывают **рассинхрон аудио/видео** на длинных файлах (>5 мин). Проверено и подтверждено как нерабочий подход.

### 1.3 НИКОГДА не писать кастомные скрипты для нарезки
Использовать ТОЛЬКО проверенные инструменты: `auto-editor` для тишины, `ffmpeg trim+concat` для ручных разрезов. Самодельные скрипты давали рассинхрон, оставляли паузы и мусорный вывод.

### 1.4 Удаление тишины: auto-editor рендерит напрямую
auto-editor — единственный проверенный инструмент для удаления тишины. Дать ему рендерить — НЕ парсить его XML и рендерить через ffmpeg (проверено, результат ужасный: дыры, растяжки, оставшиеся паузы).

```bash
auto-editor <file> --margin 0.15s --video-codec h264_nvenc --video-bitrate 10M -o output.mp4
```

**КРИТИЧНО: производительность O(n²) на длинных видео!**
- 2 мин видео → 18 сек ✓
- 25 мин видео → 20-30 мин (приемлемо)
- 1 час видео → ЧАСЫ (неприемлемо)

**Поэтому: ВСЕГДА удалять тишину ПОCЦЕНЫ (до склейки), НЕ из объединённого файла.**
Это самая важная оптимизация пайплайна. Каждая сцена 1-5 мин → auto-editor справляется за секунды.

Запускать с `nohup ... &` + `disown` (не `run_in_background`) — процесс должен пережить таймаут в 2 мин.

### 1.5 Настройки кодирования — ЗАФИКСИРОВАНЫ, НЕ МЕНЯТЬ

**Видео (NVENC):**
```
-c:v h264_nvenc -preset p4 -cq 20 -pix_fmt yuv420p
```

**Аудио (всегда 256k):**
```
-c:a aac -b:a 256k
```
256k AAC минимизирует потери при многократном ре-кодировании. Никогда не использовать 128k или 192k.

**Полная строка (copy-paste):**
```
-c:v h264_nvenc -preset p4 -cq 20 -pix_fmt yuv420p -c:a aac -b:a 256k
```

**auto-editor (зафиксированные настройки):**
```
auto-editor <file> --edit audio:0.13 --margin 0.05s,0.12s --video-codec h264_nvenc --video-bitrate 10M --audio-bitrate 256k -o <output>
```
- `--edit audio:0.13` — сбалансированный порог: режет паузы без "роботизации" речи
  - 0.15 — слишком агрессивно, слова звучат отрывисто
  - 0.08 — слишком мягко, оставляет много "эм" и пауз
  - 0.12-0.13 — естественный поток при убранном мёртвом воздухе
- `--margin 0.05s,0.12s` — асимметричный: 50мс до слова (плотный вход), 120мс после слова (сохраняет окончания)
  - До: защищает начальные согласные (п, т, к)
  - После: защищает хвостовые звуки (с, ш, ч) — без этого слова "рубятся"
  - Профессиональное правило: всегда БОЛЬШЕ padding после, чем до
- `--audio-bitrate 256k` — минимизирует артефакты AAC ре-кодирования

**ffmpeg-normalize (YouTube -14 LUFS, ТОЛЬКО последний шаг):**
```
ffmpeg-normalize <file> -o <output> -t -14 -tp -1.5 -c:a aac -b:a 256k
```

### 1.6 ВСЕГДА `-pix_fmt yuv420p`
xfade и некоторые фильтры выводят yuv444p → Windows Media Player не воспроизводит. Всегда принудительно указывать yuv420p.

### 1.7 ВСЕ команды монтажа → запускать в фоне
**КАЖДАЯ команда ffmpeg, ffmpeg-normalize и auto-editor ДОЛЖНА использовать фоновый запуск.**
Это позволяет Claude отвечать на вопросы пока идёт обработка.

- `ffmpeg-normalize` → фон (30-60 сек на сцену)
- `auto-editor` → фон (10-120 сек на сцену)
- `ffmpeg` (trim, concat, overlay) → фон (10-300 сек)
- Несколько сцен: запускать 2-3 параллельно, проверять результаты по уведомлению

Никогда не использовать `| tail -5` на длинных командах — блокирует Claude и скрывает прогресс.

### 1.8 Транскрибировать перед нарезкой
Всегда `transcribe.py` для нахождения точных границ слов. Точки разреза = конец слова + буфер 0.15с (никогда не в середине слова).

## 2. Пайплайн многосценового монтажа

**Проверенный воркфлоу для сборки видео из нескольких OBS-сцен:**

### Ключевая терминология:
- **Сцена** = одна готовая единица в папке ready/ (например `scene13_final.mp4`). Одна сцена = один тематический сегмент финального видео.
- **Многочастная сцена** = одна сцена, записанная несколькими дублями (например scene7 = `scene7.mkv` + `scene7.2.mkv` + `scene7.3.mkv`). Каждая ЧАСТЬ проходит полный пайплайн ИНДИВИДУАЛЬНО, затем части склеиваются в scene_final.mp4.
- **Простая сцена** = одна сцена из одного файла записи. Проходит пайплайн как есть.
- **Финальная склейка** = объединение ВСЕХ готовых сцен в финальное видео для YouTube. Это Шаг 6.

**КРИТИЧЕСКОЕ РАЗЛИЧИЕ:** "склеить части многочастной сцены" (Шаг 5b) ≠ "финальная склейка всех сцен" (Шаг 6). Части склеиваются ПОСЛЕ auto-editor. Сцены склеиваются ПОСЛЕ готовности всех сцен.

### Шаг 1: Обрезать каждую сцену
- Транскрибировать для точных границ слов
- Использовать `ffmpeg -i <input> -t <end> ...NVENC... <output>` или `ffmpeg -i <input> -ss <start> ...NVENC... <output>`
- ВСЕГДА ре-кодировать (NVENC), НИКОГДА не `-c copy`

### Шаг 2: Нормализовать аудио до -14 LUFS ⚠️ ДО auto-editor!
**КРИТИЧНО: Нормализовать ДО удаления тишины, НЕ после.**
Без нормализации тихие сцены получают чрезмерную нарезку auto-editor (порог воспринимает тихую речь как тишину).
Нормализация выравнивает все сцены → порог 0.13 работает стабильно по всем сценам.

```bash
ffmpeg-normalize scene.mp4 -o scene_norm.mp4 -t -14 -tp -1.5 -c:a aac -b:a 256k
```

- Сцены с дрейфом громкости (Windows auto-gain): исправить ДО нормализации:
  ```bash
  ffmpeg -y -i input.mp4 -af "volume=-4dB:enable='gte(t,240)'" -c:v copy -c:a aac -b:a 256k output.mp4
  ```
- Проверка: `ffmpeg -i <file> -af loudnorm=print_format=summary -f null -` → должно показать ~-14 LUFS

### Шаг 3: Убедиться в единообразии формата
- ВСЕ сцены должны совпадать: разрешение, fps, pix_fmt, кодек
- Если любая сцена отличается (например 1440p30 vs 1080p60): ре-кодировать с `-vf "scale=1920:1080" -r 60`
- Проверка перед склейкой: `ffprobe -select_streams v -show_entries stream=width,height,r_frame_rate,pix_fmt`

### Шаг 4: Вырезать конкретные фразы (если нужно)
- Транскрибировать нужный участок для точных таймстампов
- Использовать `ffmpeg -filter_complex` с `trim/atrim + concat`:
```bash
ffmpeg -i input.mkv -filter_complex \
  "[0:v]trim=0:END1,setpts=PTS-STARTPTS[v1]; \
   [0:v]trim=START2,setpts=PTS-STARTPTS[v2]; \
   [0:a]atrim=0:END1,asetpts=PTS-STARTPTS[a1]; \
   [0:a]atrim=START2,asetpts=PTS-STARTPTS[a2]; \
   [v1][a1][v2][a2]concat=n=2:v=1:a=1[vout][aout]" \
  -map "[vout]" -map "[aout]" \
  -c:v h264_nvenc -preset p4 -cq 20 -pix_fmt yuv420p \
  -c:a aac -b:a 256k output.mp4
```

### Шаг 5: Удалить тишину ПОCЦЕНЫ (auto-editor) ⚠️ ДО склейки!
**КРИТИЧНО: Удалять тишину из каждой сцены ИНДИВИДУАЛЬНО, не из объединённого файла.**
auto-editor имеет O(n²) производительность — 2 мин сцена = 18 сек, но 25 мин объединённый = 30+ мин.
Обработка каждой сцены по отдельности: суммарно ~2-3 мин. Обработка объединённого: 30+ мин.

```bash
# Для каждой сцены (1-5 мин каждая → секунды на сцену):
auto-editor scene1_norm.mp4 --edit audio:0.13 --margin 0.05s,0.12s --video-codec h264_nvenc --video-bitrate 10M --audio-bitrate 256k -o scene1_final.mp4
```

Если работаешь с уже объединённым файлом (нет выбора), используй `nohup ... & disown` и жди 20-30 мин.

### Шаг 5b: Склеить части МНОГОЧАСТНЫХ сцен ⚠️ ПОСЛЕ auto-editor!
**Для сцен, записанных несколькими дублями.**

**НИКОГДА не склеивать части до auto-editor. Обработать КАЖДУЮ ЧАСТЬ через Шаги 1-5 ИНДИВИДУАЛЬНО, затем склеить ГОТОВЫЕ части.**

```
ПРАВИЛЬНЫЙ порядок для многочастных сцен:
  Часть A: trim → normalize → auto-editor → partA_final.mp4
  Часть B: trim → normalize → auto-editor → partB_final.mp4
  Часть C: trim → normalize → auto-editor → partC_final.mp4
  Затем: concat partA_final + partB_final + partC_final → scene_final.mp4

НЕПРАВИЛЬНО (НИКОГДА ТАК):
  trim A, trim B, trim C → concat → normalize → auto-editor
  ❌ auto-editor на concat = O(n²), артефакты в точках склейки
```

**Команда склейки для многочастных сцен:**
```bash
printf "file 'partA_final.mp4'\nfile 'partB_final.mp4'\n..." > concat_scene.txt
ffmpeg -f concat -safe 0 -i concat_scene.txt \
  -c:v h264_nvenc -preset p4 -cq 20 -pix_fmt yuv420p \
  -c:a aac -b:a 256k \
  scene_final.mp4
```

### Шаг 6: Финальная склейка — оптимизировано для YouTube (ЗАФИКСИРОВАНО)
**Все сцены должны быть индивидуально обработаны (Шаги 1-5) перед склейкой.**

**Апскейл до 1440p — ДЕФОЛТ** — YouTube присваивает кодек VP9/AV1 вместо AVC1, значительно лучшее качество для зрителей.

```bash
# Создать список файлов
printf "file '%s'\n" scene1_final.mp4 scene2_final.mp4 ... > concat.txt

# Финальная склейка с апскейлом до 1440p
ffmpeg -f concat -safe 0 -i concat.txt \
  -vf "scale=2560:1440:flags=lanczos" \
  -c:v h264_nvenc -preset p7 -tune hq \
  -profile:v high -level 5.1 \
  -rc cbr -b:v 24M \
  -g 120 -bf 2 \
  -spatial-aq 1 -temporal-aq 1 -aq-strength 10 \
  -rc-lookahead 32 \
  -pix_fmt yuv420p \
  -colorspace bt709 -color_primaries bt709 -color_trc bt709 \
  -movflags +faststart \
  -c:a aac -b:a 384k -ar 48000 \
  FINAL.mp4
```

**Почему такие настройки:**
- `-rc cbr -b:v 24M` — YouTube рекомендация для 1440p60. CBR держит стабильные 24 Мбит/с даже на статичном "говорящей голове". VBR/CQ давали только 8 Мбит/с → YouTube снижает качество
- `-preset p7 -tune hq` — максимальное качество NVENC, финальный файл не спешит
- `-profile:v high -level 5.1` — H.264 Level 5.1 обязателен для 2560x1440@60fps. Level 4.2 поддерживает только до 1920x1080 → ошибка "Invalid Level" на 1440p
- `-g 120` — GOP 2 секунды при 60fps, YouTube стандарт для быстрой DASH/HLS-сегментации
- `-bf 2` — B-кадры для эффективности сжатия
- `-spatial-aq 1 -temporal-aq 1 -aq-strength 10` — адаптивное квантование: сохраняет детали лица и стабилизирует плоские фоны
- `-colorspace bt709 ...` — явные SDR-теги цвета, предотвращают неправильную интерпретацию YouTube
- `-movflags +faststart` — moov atom в начале файла, YouTube начинает обработку во время загрузки
- `-c:a aac -b:a 384k -ar 48000` — высококачественный аудио источник для ре-кодирования YouTube
- Апскейл 1440p → YouTube присваивает VP9/AV1 (даже для зрителей в 1080p источник VP9 = чётче)

**Ожидаемый вывод:** ~24 Мбит/с, ~13 ГБ для 75-минутного видео.

**НЕ использовать:**
- `-fflags +genpts+igndts` — может вызвать A/V рассинхрон с B-кадрами
- `-c copy` — замороженные кадры
- `-level 4.2` с 1440p — ошибка Invalid Level, использовать 5.1+
- VBR/constqp/CQ без целевого битрейта — даёт 8 Мбит/с на "говорящей голове"

### Шаг 6b: Проверка громкости перед финальной склейкой
**ВСЕГДА проверять все сцены перед финальной склейкой.** Ожидаемо: от -13.7 до -14.1 LUFS (отклонение <0.4 дБ).

```bash
# Проверить LUFS всех сцен (запустить в фоне — ~5 мин для 25 сцен)
cd scenes_for_concat && for f in *.mp4; do
  lufs=$(ffmpeg -hide_banner -i "$f" -af loudnorm=print_format=summary -f null - 2>&1 | grep "Input Integrated" | sed "s/.*: *//")
  printf "%-25s %s\n" "$f" "$lufs"
done
```

**Альтернатива (точнее, с пиками):**
```bash
for f in *.mp4; do
  result=$(ffmpeg -i "$f" -af "loudnorm=print_format=json" -f null /dev/null 2>&1 | python3 -c "
import sys, json
text = sys.stdin.read()
start = text.rfind('{')
end = text.rfind('}') + 1
if start >= 0:
    d = json.loads(text[start:end])
    print(f\"{d['input_i']} {d['input_tp']}\")
")
  lufs=$(echo "$result" | awk '{print $1}')
  peak=$(echo "$result" | awk '{print $2}')
  printf "%-25s LUFS: %7s  Peak: %7s\n" "$f" "$lufs" "$peak"
done
```

**НЕ использовать `grep -P`** — Git Bash на Windows не поддерживает Perl regex. Использовать `sed` или `python3` для парсинга.

### Шаг 7: Финальная нормализация аудио — НЕ НУЖНА
Нормализация аудио делается ПОCЦЕНЫ на Шаге 2 (до auto-editor). Все сцены уже -14 LUFS. Дополнительная нормализация финального файла не нужна.

**Если нужна (исключительный случай):**
```bash
ffmpeg-normalize final.mp4 -o output.mp4 -t -14 -tp -1.5 -c:a aac -b:a 384k
```

## 3. Правила работы с аудио

### Нормализация
- **loudnorm** (хорошо): поднимает средний уровень равномерно, минимальное усиление шума
- **Компрессор** (плохо для пост-обработки): накачивает шум между словами, делает тихие части громкими
- **afftdn** (плохо): создаёт эффект "бочки" / подводного звука на чистых записях
- YouTube нормализует ВНИЗ (громкое → -14 LUFS) но НИКОГДА не ВВЕРХ (тихое остаётся тихим)
- Записывать на -12 до -6 дБ пики → loudnorm до -14 LUFS → чистый результат

### Выравнивание громкости нескольких сцен
- Сначала измерить все сцены, найти среднее (~-21 LUFS типично для OBS)
- Поднять тихие сцены простым фильтром `volume=XdB`
- Для сцен с дрейфом Windows auto-gain: использовать линейную огибающую громкости
- Выравнивание делать ДО склейки
- loudnorm делать ПОСЛЕ всех разрезов и редактирования (последний шаг)

### Каскадная деградация AAC — КРИТИЧНО
- Каждый цикл кодирования/декодирования AAC безвозвратно теряет качество
- Повышение битрейта выше 256k НЕ помогает — потерянная информация не восстанавливается
- **256k AAC — практический потолок для речи** (прозрачный при 128k однопроходный)
- **Для будущих видео:** использовать WAV/PCM как промежуточный формат, кодировать AAC только ОДИН раз на финальном шаге:
  ```
  OBS (AAC) → извлечь WAV → всю обработку в WAV → финальный AAC 256k (один раз)
  ```

### Проблемы Windows с аудио
- **Режим коммуникации** тихо снижает громкость микрофона во время записи → отключить в Панель управления → Звук → Связь → "Не выполнять никаких действий"
- **Auto gain** постепенно меняет уровень микрофона → отключить в настройках звука → свойства устройства ввода

## 4. Справочник инструментов

### Python (папка tools/ в рабочей директории)
```
transcribe.py <file>                        → пословесный JSON транскрипт (large-v3, CUDA)
detect_scenes.py <file>                     → JSON границ сцен
concat_clips.py f1 f2 [--transition fade]   → склейка клипов с переходами
overlay_image.py <video> <image> [opts]     → наложение изображения с fade
build_srt.py <transcript.json>              → SRT файл субтитров
verify_output.py <video>                    → проверка целостности вывода
```

### C5. Текстовые оверлеи / плашки

**Дерево решений — какой подход использовать:**
```
Только текст (без emoji, без зачёркивания)    → Подход D: drawtext с шейком
Текст + цветные emoji (🥲, 🙉, etc.)          → Подход E: Pillow PNG + overlay с шейком
Зачёркнутый текст                             → Подход E: Pillow PNG (drawtext не умеет strike)

```

**КРИТИЧЕСКИЕ УРОКИ:**
- `drawtext` НЕ МОЖЕТ рендерить цветные emoji — только монохромные глифы. Используй Pillow + `seguiemj.ttf` + `embedded_color=True`
- `drawtext` НЕ МОЖЕТ делать зачёркивание — используй Pillow `draw.line()`
- `amix` с несколькими входами по умолчанию **делит громкость на N**! ВСЕГДА используй `normalize=0`
- Все оверлеи можно объединить в ОДНОЙ ffmpeg команде (несколько drawtext + несколько overlay входов)
- Сначала тестировать на одном кадре (Pillow composite), потом рендерить видео — намного быстрее итерации

### C7. Овальная/круглая маска вебкамеры

**Шаг 1: Найти точные координаты прямоугольника вебкамеры (анализ пикселей Pillow)**

Положение прямоугольника вебкамеры варьируется между записями. НИКОГДА не угадывать координаты — всегда определять автоматически.

```python
from PIL import Image
import numpy as np

img = np.array(Image.open('frame_1080p.png'))

# Найти ВЕРХНИЙ край: сканировать строки для резкого скачка яркости в диапазоне x вебкамеры
for y in range(800, 950):
    above = int(np.mean(img[y-3:y, 50:300, :]))
    below = int(np.mean(img[y:y+3, 50:300, :]))
    if abs(above - below) > 30:
        print(f'Верхний край: y={y}')
        break

# Найти ПРАВЫЙ край: сканировать столбцы для резкого падения яркости в диапазоне y вебкамеры
for x in range(250, 450):
    left_val = int(np.mean(img[950:1050, x-2:x, :]))
    right_val = int(np.mean(img[950:1050, x:x+2, :]))
    if abs(left_val - right_val) > 30:
        print(f'Правый край: x={x}')
        break

# ЛЕВЫЙ край: обычно x=0. НИЖНИЙ край: обычно y=высота_кадра (край).
```

**Шаг 1b: Визуальная верификация с цветными контурами**
```python
from PIL import Image, ImageDraw
img = Image.open('frame_1080p.png').copy()
draw = ImageDraw.Draw(img)
# КРАСНЫЙ = обнаруженный прямоугольник вебкамеры, ЗЕЛЁНЫЙ = вписанный овал
draw.rectangle([cam_x, cam_y, cam_x+cam_w, cam_y+cam_h], outline='red', width=3)
cx, cy = cam_x + cam_w//2, cam_y + cam_h//2
rx, ry = cam_w//2 - 4, cam_h//2 - 4
draw.ellipse([cx-rx, cy-ry, cx+rx, cy+ry], outline='lime', width=3)
img.crop((0, cam_y-50, cam_w+100, 1080)).save('debug_oval.png')
```

**Шаг 2: Применить овальную маску через ffmpeg**
```bash
# Переменные: CAM_X, CAM_Y, CAM_W, CAM_H = обнаруженный прямоугольник вебкамеры
# CX=CAM_W/2, CY=CAM_H/2, RX=CX-4, RY=CY-4
ffmpeg -y -i input.mp4 -filter_complex "
[0:v]scale=1920:1080,split[main1][main2];
[main2]crop=CAM_W:CAM_H:CAM_X:CAM_Y,format=yuva420p,geq=lum='p(X,Y)':cb='p(X,Y)':cr='p(X,Y)':a='if(lt(pow(X-CX,2)/pow(RX,2)+pow(Y-CY,2)/pow(RY,2),1),255,0)'[oval];
[main1]drawbox=x=CAM_X:y=CAM_Y:w=CAM_W:h=CAM_H:c=0x111111:t=fill[bg];
[bg][oval]overlay=CAM_X:CAM_Y" \
-c:v h264_nvenc -preset p4 -cq 20 -pix_fmt yuv420p -c:a aac -b:a 256k -r 60 output.mp4
```

**Как это работает:**
1. `split` → две копии видео
2. Копия 2: `crop` извлекает вебкамеру → `geq` формула эллипса делает углы прозрачными
3. Копия 1: `drawbox` заполняет прямоугольник вебкамеры тёмным фоном
4. `overlay` возвращает овальную вебкамеру → углы показывают тёмный drawbox, центр показывает лицо

## 6. Известные проблемы и антипаттерны

### ФАТАЛЬНЫЕ — вызовут видимые проблемы:
1. **`-c copy` при разрезке** → замороженные кадры (проблема ключевых кадров). ВСЕГДА ре-кодировать.
2. **Фильтры select/aselect** → аудио/видео рассинхрон на длинных файлах. НИКОГДА не использовать.
3. **Самодельные скрипты нарезки** → рассинхрон, пропущенные паузы, мусор. Использовать auto-editor.
4. **afftdn шумоподавление** → эффект "бочки" / подводного звука на чистых записях. Не использовать.
5. **Компрессор для нормализации** → накачивает шум между словами. Использовать loudnorm.
6. **`| tail -N` на длинных командах** → скрывает прогресс, выглядит зависшим. Использовать фоновый запуск.
7. **Смешивание fps в concat** (30fps + 60fps с `-c copy`) → дублирующиеся/замороженные кадры. Сначала ре-кодировать.

### ПРЕДУПРЕЖДЕНИЯ:
8. **bash exit 127**: Длинные python команды → перезапустить сессию или использовать powershell.exe.
9. **Требуется CUDA Toolkit**: `cublas64_12.dll` нужен CUDA 12.x.
10. **large-v3 ~3 ГБ**: Загружается при первом запуске. Кэшируется в `~/.cache/huggingface/hub/`.
11. **xfade → yuv444p**: ВСЕГДА добавлять `-pix_fmt yuv420p`. Без этого ошибка воспроизведения.
12. **AAC ре-кодирование**: Не ре-кодировать AAC больше 2 раз. Нормализовать каждый источник отдельно.
13. **Windows auto-gain**: Постепенно меняет уровень микрофона. Отключить в настройках звука.

## 7. Финальный рендер: оверлеи + микширование аудио — ОДИН ПРОХОД

### 7.0 ЗОЛОТОЕ ПРАВИЛО: Всё в ОДНОЙ ffmpeg команде
**ВСЕ оверлеи (текст, emoji, изображения) + ВСЁ аудио (музыка, SFX, дакинг) = ОДИН проход рендера.**
Несколько `-i` входов — это НЕ несколько рендеров — ffmpeg читает их все одновременно.
Это избегает каскадной потери качества от ре-кодирования.

### 7.1 КРИТИЧНО: `amix normalize=0`
**ВСЕГДА использовать `normalize=0` с amix.** Без этого amix делит каждый вход на 1/N — голос теряет 50-66% громкости.

### 7.2 Воркфлоу: подготовка финального рендера

**Шаг 1: Собрать все запросы оверлеев**
- Получить точные таймкоды (ММ:СС → конвертировать в секунды)
- Получить текст, длительность, тип (простой текст / emoji / изображение)
- Построить таблицу: # | таймкод | секунды | текст | длительность | метод

**Шаг 2: Создать PNG оверлеи через Pillow (для emoji/изображений)**
- Холст ДОЛЖЕН совпадать с разрешением видео (например, 2560x1440 для 1440p видео)
- Использовать `stroke_width` + `stroke_fill` (НЕ петли offset-copy) — совпадает с видом drawtext
- Сначала тестировать на извлечённом кадре (Pillow composite preview — мгновенно)
- Полный рецепт в C5 Подход E

**Шаг 3: Тест на клипе 30-40 секунд**
- Выбрать сегмент с хотя бы одним drawtext И одним PNG оверлеем
- Включить микширование аудио для проверки дакинга + normalize=0
- Показать пользователю, получить одобрение

**Шаг 4: Запустить полный рендер в фоне**
- `nohup ffmpeg ... > render_log.txt 2>&1 & disown`
- Оставаться доступным для вопросов пока идёт рендер
- Ожидаемая скорость: ~0.8-1.0x для 1440p CBR 24M с оверлеями

### 7.3 Шаблон структуры команды

```bash
ffmpeg -y \
  -i video.mp4 \
  -i overlay1.png -i overlay2.png ... \        # PNG оверлеи
  -i stinger.mp3 \                              # SFX
  -stream_loop N -i music.mp3 \                 # зацикленные треки
  -filter_complex "
    [0:v]                                        # ВИДЕО ЦЕПОЧКА:
    drawtext=...enable='between(t,S1,E1)',       # простой текст (через запятые)
    drawtext=...enable='between(t,S2,E2)'
    [td];                                        # → выход после всех drawtext
    [td][1:v]overlay=SHAKE:enable='...'[ov1];    # PNG оверлеи (последовательная цепочка)
    [ov1][2:v]overlay=SHAKE:enable='...'[ov2];
    [ov2][3:v]overlay=0:0:enable='...'[vout];    # изображения без шейка

    [0:a]VOICE_PROCESSING,asplit=2[speech][sc];  # АУДИО ЦЕПОЧКА:
    [N:a]TRACK_PROCESSING[track1];               # каждый трек: громкость, fade, задержка
    [M:a]BG_VOLUME_AUTOMATION[bg];               # фон: сложная огибающая громкости
    [bg][sc]sidechaincompress=...[bg_ducked];     # дакинг только фона
    [speech][track1][bg_ducked]...amix=inputs=N:normalize=0:duration=first[aout]
  " \
  -map "[vout]" -map "[aout]" \
  -c:v h264_nvenc ... -c:a aac -b:a 384k \
  output.mp4
```

### 7.4 Стиль drawtext с шейком (ЗАФИКСИРОВАНО — шрифт Impact, янтарный цвет)
```
drawtext=fontfile='C\:/Windows/Fonts/impact.ttf':\
  fontsize=110:fontcolor=0xFFBB00:\
  borderw=3:bordercolor=0x000000:\
  shadowx=2:shadowy=2:shadowcolor=0x000000@0.8:\
  x=(w-tw)/2+3*sin(t*15):\
  y=(h-th)/2-h*0.15+3*cos(t*12):\
  text='ТЕКСТ':enable='between(t,START,END)'
```
- **Шрифт:** Impact (тестировали Impact, Tahoma Bold, Montserrat — выбран Impact)
- **Цвет:** янтарный 0xFFBB00 с **чёрной** обводкой stroke=3 (тестировали красный, жёлтый, янтарный)
- **Позиция:** центр по горизонтали, **15% выше центра** по вертикали
- **Шейк:** `3*sin(t*15)` горизонтально, `3*cos(t*12)` вертикально
- **Цепочка нескольких:** разделять запятыми, НЕ точка с запятой между drawtext фильтрами

### 7.5 PNG оверлей с шейком
```
[prev][N:v]overlay=3*sin(t*15):3*cos(t*12):enable='between(t,START,END)'[next]
```
- Для изображений БЕЗ шейка: `overlay=0:0:enable='...'`
- Холст PNG ДОЛЖЕН совпадать с разрешением видео
- Цепочка последовательно: `[ov1]→[ov2]→[ov3]→...→[vout]`

### 7.6 Параметры микширования аудио (рекомендуемые значения)

| Параметр | Значение | Почему |
|---|---|---|
| Фоновая музыка | **0.078 (7.8%)** | Ненавязчивая атмосфера |
- Вступительная музыка | 0.14 (14%) | Создаёт настроение, затихает перед фоновой |
| Кинематографическая вставка | 0.19 (19%) | Намеренный музыкальный момент |
| Перкуссия/удар | 0.135 (13.5%) | Энергия для экшн-секций |
| Финальная музыка | 0.19 (19%) | Завершающий аккорд |
| **Фон во время вставок** | **0% (ВЫКЛ)** | **КРИТИЧНО: фоновая музыка ДОЛЖНА быть полностью тихой когда играет любая другая музыка** |
| Дакинг ratio | **1.8** | Мягкий — музыка чуть ниже во время речи, не исчезает |
| Дакинг threshold | 0.02 | Срабатывает на голосе |
| Дакинг attack | 300ms | Плавное затухание |
| Дакинг release | 2500ms | Медленный возврат — ощущается естественно |
| `amix normalize=0` | **КРИТИЧНО** | Предотвращает потерю громкости голоса |
| `duration=first` | Длина видео | Все треки обрезаются по длине видео |

### 7.7 Автоматизация громкости фоновой музыки
Когда фоновая музыка должна затихать для вставок и возвращаться:
```
volume='if(lt(t,T_START),0,
  if(lt(t,T_FADEIN_END),VOL*(t-T_START)/FADE_DUR,
  if(lt(t,T_FADEOUT_START),VOL,
  if(lt(t,T_FADEOUT_END),VOL*(T_FADEOUT_END-t)/FADE_DUR,
  0))))':eval=frame
```
Вложить несколько `if()` для нескольких секций.

### 7.8 Усиление голоса (компенсация Windows auto-gain)
```
volume='if(lt(t,4),1.6,if(lt(t,10),1.3,1.0))':eval=frame
```
Применить ДО asplit. Только если запись имеет дрейф громкости от Windows auto-gain.

### 7.9 adelay для тайминговых аудио вставок
```
[N:a]adelay=MILLISECONDS|MILLISECONDS,volume=VOL,afade=...,atrim=0:END,asetpts=PTS-STARTPTS[name]
```
- Значение `adelay` в **миллисекундах** (120000 = 120 секунд)
- `adelay` ДОЛЖЕН быть ДО `atrim` — иначе atrim режет до применения задержки
- Значения `afade st=` абсолютные (включают смещение задержки)

### 7.10 Ошибки которые мы допустили (учись на них)

| Ошибка | Что произошло | Исправление |
|---|---|---|
| `amix` без `normalize=0` | Голос потерял 10+ дБ | Всегда `normalize=0` |
| Pillow offset-copy обводки | Выглядело иначе чем drawtext | Использовать `stroke_width` + `stroke_fill` |
| PNG 1920x1080 на 1440p видео | Оверлеи слишком маленькие, неправильная позиция | Холст ДОЛЖЕН совпадать с разрешением видео |
| drawtext цветные emoji | Чёрные/монохромные глифы | Pillow PNG + seguiemj.ttf + `embedded_color=True` |
| drawtext зачёркивание | Невозможно | Pillow `draw.line()` |
| `atrim` до `adelay` | Аудио заканчивалось раньше | Всегда `adelay` → `atrim` |
| Дакинг ratio=6 | Музыка полностью исчезала | Снижен до 1.8 |
| Тест оверлеев на полном видео | Медленная итерация (минуты на попытку) | Тест на одном кадре (Pillow composite — мгновенно) |
| Хирургическая вставка с `-c copy` | Замороженные кадры в точках склейки | Всегда ре-кодировать через NVENC |

### 7.11 НЕ делать:
- Использовать `normalize=1` (дефолт amix) — убивает громкость голоса
- Использовать `ratio=6` для дакинга — музыка полностью исчезает
- Использовать музыку с вокалом — конкурирует с речью
- Пропускать `afade` на музыке — резкое начало/конец звучит плохо
- Использовать `-c:v copy` при изменении аудио — может вызвать замороженные кадры с `-ss`
- Рисовать цветные emoji через drawtext — использовать Pillow PNG оверлей
- Тестировать стили оверлеев на полном видео — тестировать на извлечённом кадре

## 8. Фикс True Peak — ОБЯЗАТЕЛЬНЫЙ финальный шаг

**ВСЕГДА запускать после ЛЮБОГО рендера.** Без исключений. Занимает ~5 мин, видео не трогается.

```bash
ffmpeg -y -i FINAL.mp4 \
  -c:v copy \
  -af "alimiter=limit=0.75:level=false:attack=3:release=30" \
  -c:a aac -b:a 384k -ar 48000 \
  -movflags +faststart \
  FINAL_fixed.mp4
```

**Почему:** Усиление голоса, суммирование amix и громкие музыкальные вставки создают пики True Peak выше 0 дБFS (клиппинг). Лимитер ограничивает все пики на -1.5 дБTP. Без него зрители слышат "выстрел" на согласных.

**Параметры:**
- `limit=0.841` = -1.5 дБFS потолок (10^(-1.5/20) = 0.841)
- `level=false` — КРИТИЧНО: предотвращает обратное усиление громкости лимитером
- `-c:v copy` — безопасно здесь, так как НЕТ `-ss` обрезки (полный проход файла)

**Проверка после фикса:**
```bash
ffmpeg -i FINAL_fixed.mp4 -af "loudnorm=I=-14:TP=-1.5:LRA=11:print_format=json" -f null /dev/null
```
Ожидаемо: `input_tp` ≤ -1.0 дБTP (идеально: -1.5 дБTP)

**НЕ использовать `aresample=192000`** — создаёт НОВЫЕ межвыборочные пики.
